# Spec: sdd-kit spec-forensics stage + persisted subagent join

## Purpose

sdd-kit cannot report the real, per-task cost of a whole spec run. The exec
stage records `actual_tokens` per task, but that figure is `cache_read`-blind
and undercounts real subagent spend by ~28× (issue #15), and there is no
attribution of orchestrator overhead — where ~99% of a run's cost actually
lives. Any forensics is impossible today because `execution_state.json` stores
no `agentId`/`sessionId`, so nothing links a task to its
`subagents/agent-<id>.jsonl`; the only bridge is the *soft* `meta.json.description`
label, which breaks if brief phrasing changes. This feature (a) makes exec
persist the subagent `agentId`(+`sessionId`) per task so the
`task_id → agentId → agent-<id>.jsonl → real billable` join is exact, and
(b) adds a `spec-forensics` stage that turns a finished `docs/specs/<slug>/`
into a deterministic cost report (per-task real cost incl. `cache_read`,
estimate-vs-real, orchestrator overhead, budget-pause timeline). Audience:
sdd-kit maintainers who need the true cost profile to later fix estimation and
the budget-pause. It reuses claude-token-debug's `token-cost` CLI and does not
reimplement transcript parsing.

Change type: feat

## Scope

**In scope:**
- Exec stage persists `agentId` and `sessionId` per task in
  `execution_state.json` (schema change: two new nullable fields + `required`).
- When a task's `agentId` cannot be obtained, exec records `agentId: null`,
  notes an `incidencia`, and does not abort the run.
- Exec resuming an older `execution_state.json` that lacks the new fields
  upgrades it in place (adds them as `null`) without crashing.
- A `spec-forensics` stage over a SPECDIR that produces `forensics.json` plus a
  human-readable summary, reusing the `token-cost` CLI's `analyze()`
  (`--json`/`--boundary`) output. The deterministic report is generated by a
  script (no LLM judgment); the skill is a thin wrapper.
- The report covers: per-task real tokens and cost (incl. `cache_read`),
  estimate-vs-real deviation using real tokens, orchestrator-vs-subagent split,
  and a budget-pause timeline.
- Invocation: `/sdd-kit:forensics <SPECDIR>` shortcut over a
  `forensics.mjs <SPECDIR>` CLI.

**Out of scope (non-goals):**
- Recalibrating `estimated_tokens` — that is `plan-writer-token-estimator`.
- Changing the budget-pause threshold — its own follow-up.
- Reimplementing transcript parsing — reuse the `token-cost` CLI.
- spec-forensics mutating any input: it is read-only over the SPECDIR (never
  writes `execution_state.json` or the plan).

## Functional Requirements

### R1 — Exec persists subagent agentId/sessionId per task

Depende de: —

The exec stage MUST, at each task boundary, record in
`execution_state.json` the `agentId` and `sessionId` of the subagent that
executed the task, and the state schema MUST accept those two fields (nullable)
as required properties of every task object.

#### R1.S1 — Task executed by a subagent
- GIVEN a task delegated to exactly one subagent that runs to completion
- WHEN exec writes the task-boundary state
- THEN that task object in `execution_state.json` has a non-null string
  `agentId` such that an `agent-<agentId>.jsonl` transcript is resolvable
  under the Claude projects root (`~/.claude/projects/**/subagents/`, or an
  overridable root for tests) — NOT under `SPECDIR` itself, which subagent
  transcripts never populate (confirmed empirically: real transcripts live
  alongside the orchestrator session that spawned them, keyed by the
  agent's own id, which is effectively globally unique)
- AND the task object has a `sessionId` string field (non-null when available)
- AND the state validates against `execution_state.schema.json`.

#### R1.S2 — agentId cannot be obtained for a task
- GIVEN a task whose executing subagent's `agentId` cannot be determined
- WHEN exec writes the task-boundary state
- THEN that task object has `agentId: null` and a non-null `incidencia` string
  recording the missing join
- AND the run is not aborted (exec continues to the next task per the DAG)
- AND the state validates against `execution_state.schema.json`.

#### R1.S3 — Resuming a pre-schema state
- GIVEN an existing `execution_state.json` whose task objects lack `agentId`
  and `sessionId` (e.g. produced before this change)
- WHEN exec loads it to resume
- THEN exec adds `agentId: null` and `sessionId: null` to each task object that
  lacks them, without throwing
- AND the upgraded state validates against `execution_state.schema.json`.

### R2 — Per-task real cost and estimate-vs-real report

Depende de: R1

The spec-forensics stage SHALL produce, for a SPECDIR with a completed run, a
`forensics.json` listing each task's real token consumption and cost (including
`cache_read`) resolved via `agent-<agentId>.jsonl`, and the deviation of real
tokens from `estimated_tokens`.

#### R2.S1 — Full run reported per task
- GIVEN a SPECDIR with `execution_state.json` where tasks carry non-null
  `agentId`, each resolvable to an `agent-<id>.jsonl` transcript under the
  Claude projects root (see R1.S1 — not under SPECDIR)
- WHEN `forensics.mjs <SPECDIR>` runs
- THEN `SPECDIR/forensics.json` contains a `tasks` object keyed by `task_id`,
  each with numeric `real_tokens`, `real_cost_usd`, `estimated_tokens`, and
  `deviation_real` (= `real_tokens - estimated_tokens`), where `real_tokens`
  counts `cache_read_input_tokens`
- AND a readable summary of the same per-task figures is printed to stdout.

#### R2.S2 — Task with no resolvable transcript
- GIVEN a task whose `agentId` is `null`, or whose `agent-<agentId>.jsonl`
  file cannot be found under the Claude projects root
- WHEN `forensics.mjs <SPECDIR>` runs
- THEN that task's entry in `forensics.json` has `"resolved": false` (and null
  cost/token fields) while every other task is reported with real figures
- AND the run exits non-fatally (exit code 0).

### R3 — Orchestrator overhead and budget-pause timeline

Depende de: R2

The spec-forensics stage SHALL include in `forensics.json` the
orchestrator-vs-subagent cost split and a budget-pause timeline derived from
the run.

#### R3.S1 — Split and pause reported
- GIVEN a SPECDIR whose run had a recorded `pause` and whose session transcript
  is available to the `token-cost` CLI
- WHEN `forensics.mjs <SPECDIR>` runs
- THEN `forensics.json` has an `orchestrator` object with numeric
  `real_tokens` and `real_cost_usd` and a `subagents_total` object with the
  same fields, sourced from the CLI's `orchAll`/`subTotal`
- AND `forensics.json` has a `pause_timeline` array with one entry carrying the
  pause's `at_task` and accumulated `real_tokens`.

#### R3.S2 — Run with no pause
- GIVEN a SPECDIR whose `execution_state.json` has `pause: null`
- WHEN `forensics.mjs <SPECDIR>` runs
- THEN `forensics.json` has `pause_timeline: []` and no error is raised.

### R4 — Read-only and graceful degradation

Depende de: R2

The spec-forensics stage MUST NOT modify any SPECDIR input, and MUST produce a
report flagged as incomplete rather than crashing when join data is missing.

#### R4.S1 — Inputs left byte-identical
- GIVEN a SPECDIR with `execution_state.json` and `execution_plan.json`
- WHEN `forensics.mjs <SPECDIR>` runs to completion
- THEN `execution_state.json` and `execution_plan.json` are byte-for-byte
  unchanged (same SHA-256 as before the run).

#### R4.S2 — Missing join data
- GIVEN a SPECDIR whose task objects all lack `agentId`, or whose non-null
  `agentId`s resolve to no `agent-<id>.jsonl` anywhere under the Claude
  projects root
- WHEN `forensics.mjs <SPECDIR>` runs
- THEN `forensics.json` has `"incomplete": true` and an `incomplete_reason`
  string naming the missing data
- AND the process exits with code 0 (non-fatal), without a stack trace.

### R-E2E — Finished spec to deterministic cost report

Depende de: R1, R2, R3, R4

The system SHALL take a completed SPECDIR whose exec persisted the subagent
join and produce a single deterministic `forensics.json` with per-task real
cost and deviation, orchestrator-vs-subagent split, budget-pause timeline, and
unresolved-task flags — leaving the SPECDIR inputs untouched.

#### R-E2E.S1 — Full forensics over a fixture SPECDIR
- GIVEN a fixture SPECDIR containing an `execution_state.json` (tasks with
  `agentId`, one task deliberately with `agentId: null`, a recorded `pause`)
  and synthetic `agent-*.jsonl` transcripts placed under a fixture Claude
  projects root (passed to `forensics.mjs` via its root-override option —
  mirroring `token-cost.mjs`'s `TOKEN_COST_PROJECTS_ROOT`/`projectsRoot`,
  never under the fixture SPECDIR itself)
- WHEN `/sdd-kit:forensics <SPECDIR>` (via `forensics.mjs <SPECDIR>`) runs
- THEN `SPECDIR/forensics.json` reports every resolvable task with
  `real_tokens`/`real_cost_usd`/`deviation_real`, marks the `agentId: null`
  task `"resolved": false`, includes the `orchestrator`/`subagents_total`
  split and a one-entry `pause_timeline`
- AND `execution_state.json` and `execution_plan.json` are byte-identical to
  before the run
- AND the process exits with code 0.

## Technical Requirements

- **Stack / framework:** Node.js, stdlib only, no npm deps, no network — same
  constraints as the existing `token-cost.mjs` CLI it reuses.
- **Integraciones:** Consumes claude-token-debug's `token-cost` CLI
  `analyze()` output (`session/subs/orchestrator/subTotal/orchAll`, via
  `--json`/`--boundary`); MUST NOT reimplement transcript scanning or pricing.
- **Rendimiento:** N/A (offline analysis of already-written transcripts).
- **Seguridad / privacidad:** Read-only over the SPECDIR and transcript files;
  no writes to inputs, no external calls.
- **Datos / almacenamiento:** `execution_state.schema.json` gains `agentId`
  (`["string","null"]`) and `sessionId` (`["string","null"]`) on the task
  object, both added to its `required` list. New artifact
  `SPECDIR/forensics.json` (deterministic, machine-checkable). Report
  generation is a script (no LLM), so the skill is a thin wrapper.
- **Restricciones adicionales:** Backward-compatible with pre-schema
  `execution_state.json` (R1.S3). The `token-cost` CLI and its `analyze()`
  shape are an existing dependency and are treated as stable.

## Assumptions & Open Questions

- **Assumed:** exec can obtain the executing subagent's `agentId` in the normal
  case; R1.S2 defines the fallback (`null` + `incidencia`, no abort) precisely
  so the feature is robust even if capture fails for a given task or platform.
- **Assumed:** `sessionId` is secondary context; the exact task↔transcript join
  is via `agentId` → `agent-<id>.jsonl`. `sessionId` may be `null` when not
  available without weakening the join.
- **Corrected (was wrong in the original spec, found empirically while
  executing exec-persist-agentid):** subagent transcripts do NOT live under
  `SPECDIR/subagents/` — that directory is never created there. They live
  under `~/.claude/projects/<project>/<session-uuid>/subagents/agent-<id>.jsonl`,
  i.e. sibling to whichever orchestrator session spawned them. Since agent
  ids are effectively globally-unique random hex strings, the join is a
  global search for `agent-<agentId>.jsonl` under the Claude projects root
  (mirroring `token-cost.mjs`'s own project/session resolution), not a
  SPECDIR-relative lookup. `forensics.mjs` must expose a root-override
  (mirroring `TOKEN_COST_PROJECTS_ROOT`/`projectsRoot`) so tests never touch
  the real `~/.claude/projects`.
- **Assumed:** the deterministic report core is a `forensics.mjs` script
  (mirroring `token-cost.mjs`) so every AC is `[auto]`; the `/sdd-kit:forensics`
  skill is a thin invocation wrapper.
- **Assumed:** the `token-cost` CLI `analyze()` fields
  (`orchestrator`/`subTotal`/`orchAll`) are the sanctioned source for the
  orchestrator-vs-subagent split; forensics adds the SDD-specific per-task and
  pause layers on top.
